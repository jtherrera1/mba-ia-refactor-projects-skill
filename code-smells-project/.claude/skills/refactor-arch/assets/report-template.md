# REPORT FORMAT — Architecture Audit Report

> This file defines the **expected output format** for the architecture audit report.
> Every section below is required. Keep the structure, labels, and severity tags exactly as shown.
> The example at the bottom illustrates a complete filled-in report.
> This needs to be presented in plain text format.
---

## Required Sections

### 1. Header
```
================================
ARCHITECTURE AUDIT REPORT
================================
```

### 2. Project Metadata

| Field   | Description                                      |
|---------|--------------------------------------------------|
| Project | Name of the analyzed project                     |
| Stack   | Primary language and framework (e.g. Python + Flask) |
| Files   | Number of files analyzed and approximate total lines of code |

```
Project: <project-name>
Stack:   <language> + <framework>
Files:   <N> analyzed | ~<LOC> lines of code
```

### 3. Summary

One-line count of findings grouped by severity.

```
Summary
CRITICAL: <n> | HIGH: <n> | MEDIUM: <n> | LOW: <n>
```

**Severity classification for anti-patterns:**

To standardize your audit and the reports generated by the AI, use the following classification scale based on MVC and SOLID issues:

CRITICAL: Serious architecture or security failures that prevent proper functioning, expose sensitive data (e.g., hardcoded credentials, SQL Injection), or completely violate separation of responsibilities (e.g., a “God Class” containing database access, complex business logic, and routing in the same file).

HIGH: Major violations of the MVC pattern or SOLID principles that make maintenance and testing very difficult (e.g., heavy business logic trapped inside Controllers, strong coupling without Dependency Injection, or use of mutable global state across the entire application).

MEDIUM: Standardization issues, code duplication, or moderate performance bottlenecks (e.g., N+1 database queries, improper middleware usage, missing route validations).

LOW: Readability improvements, poor variable naming, or loose “magic numbers” scattered throughout the code.

### 4. Findings

One block per finding, ordered by severity (CRITICAL first).

```
[<SEVERITY>] <Anti-pattern Name>
File: <filename>:<lines>
Description: <what is wrong>
Impact: <consequence of leaving it unaddressed>
Recommendation: <how to fix it>
```

### 5. Footer

```
================================
Total: <N> findings
================================
```

ARCHITECTURE AUDIT REPORT
================================
Project: code-smells-project
Stack:   Python + Flask
Files:   4 analyzed | ~800 lines of code

Summary
CRITICAL: 4 | HIGH: 5 | MEDIUM: 2 | LOW: 3

Findings

[CRITICAL] God Class / God Method
File: models.py:1-350
Description: Arquivo único contém toda lógica de negócio, queries SQL,
             validação e formatação para 4 domínios diferentes.
Impact: Impossível testar em isolamento, qualquer mudança afeta tudo.
Recommendation: Separar em models e controllers por domínio.

File: app.py:8
Description: SECRET_KEY hardcoded como 'minha-chave-super-secreta-123'
...
